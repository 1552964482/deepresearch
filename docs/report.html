DeepResearch-MultiAgent

面向复杂研究任务的多智能体协作系统 —— 在弱基座 LLM 上通过工程化 Agent 编排产出可追溯、带引用、可量化评测的研究报告

Python 3.11+ · asyncio mimo-v2.5-pro × 4 keys Judge: GPT-5.4 (独立) bge-small-zh · SQLite 向量 Tavily 检索 65 单元测试

1 · 项目概览

给定一个研究问题（如「RAG 检索增强生成的核心流程与常见优化手段」），系统自动完成 规划 → 检索 → 阅读压缩 → 写作 → Red-Blue 对抗评审的全链路协作，输出一份带引用、结构化、 并经过自动化评测的研究报告。

要解决的三个痛点

① 没有引用

LLM 直答的结论不带来源，无法追溯，可信度低。

② 覆盖不全

单次生成容易漏掉关键侧面（如 RLHF 漏讲 KL 惩罚）。

③ 难以量化

「报告写得好不好」缺乏客观、可重复、有统计显著性的度量。

核心设计立场

  • 弱基座 + 强工程：主链路用 mimo-v2.5-pro（弱基座），靠 Agent 编排把质量做上去，而不是堆模型。
  • 可追溯：每条断言都通过 [n] 标记追溯到具体来源 URL。
  • 评测合法性优先：Judge 使用独立模型族（GPT-5.4），避免 self-preference bias。
  • 实证而非吹嘘：所有简历数字都能从 experiments/results/ 复现。

4 · LLM 网关 MimoPool

mimo 给了 4 个 API key，每个 100 RPM、安全并发 4。最朴素的 round-robin 在突发流量下会把单 key 打爆触发 429。 MimoPool 做了三件事来用好这 4 个 key。

① 每 key 双重限流

每个 key 持有：

  • asyncio.Semaphore(4) —— 硬并发上限
  • SlidingWindowBucket(rpm=100) —— 60 秒滑动窗口，按时间戳队列实现的令牌桶

一次请求必须同时占住这两个槽位才能发出去。

② Least-in-flight 选 key

每次请求按 (in_flight ASC, recent_rpm_usage ASC) 排序，挑当前最闲的 key，避免短时扎堆。

def _pick_slot(self, exclude=None):
    candidates = [s for s in self._slots
                  if s.idx not in exclude and s.available]
    # 最少在飞优先，并列时选近 1 分钟用量更少的 key
    candidates.sort(key=lambda s: (s.in_flight, s.bucket.usage()))
    return candidates[0] if candidates else None

③ 三段重试 + 指数退避

实测

35 题完整 pipeline 跑下来，4 keys 实际最高 RPM 用量仅 1-3 / 100。瓶颈不在 RPM，而在单次 mimo 调用的延迟（平均 20-40s）；16 路并发对当前任务规模绰绰有余。每个 key 的 in_flight / 成功率 / 平均延迟 / RPM 用量都通过 pool.stats() 实时暴露。

5 · 独立 Judge 与降级保护

评测器必须独立于被测模型族，否则会有系统性的 self-preference bias （Zheng et al. 2023 Judging LLM-as-a-Judge；Panickssery et al. 2024 LLMs Cannot Self-Reward）。 本项目 Judge 走完全独立的端点（aveve.xyz 上的 GPT-5.4），从 model family 到 base_url 都和被测分开。

5 维度 × 自一致性

Judge 对每份报告打 5 个维度（准确性 / 完整性 / 逻辑性 / 引用质量 / 可读性），每维度 1-5 分。 同一份报告跑 n_samples 次取均值降低 Judge 自身方差。

降级保护（真实工程问题）

现场

评测跑到一半，aveve.xyz 的 /chat/completions 全部返回 502 Bad Gateway，三次重试都挂。

解决方式：JudgeClient 加 fallback_pool 参数，独立端点不可用时降级到 mimo，但每条 score 打标：

return JudgeScore(
    ...,
    backend=("mimo-fallback" if used_fallback else self.config.model),
    self_bias_risk=used_fallback,   # 评测合法性可追溯
)

Eval summary 头部会明确写出，绝不让降级数据被误读为独立 Judge 结果：

- judge backends seen: mimo-fallback×35
- ⚠️  self-bias risk: 35 samples scored via mimo fallback

注：本报告中所有 judge_overall 数据均来自 fallback 路径；两个不依赖 Judge 的规则指标 （factual_accuracy、citation_coverage）的效应量不受此影响，是更可靠的结论来源。

6 · 编排：状态机 + DAG

没有用 LangGraph，而是自研一个表驱动状态机 + 异步 DAG。所有合法迁移声明在一个 TRANSITIONS 字典里，非法迁移直接抛 InvalidTransition，让误用快速失败。

ResultEnvelope：异常隔离

每个 Agent 返回 ResultEnvelope[T] 而不是裸值/裸异常。一个并发分支抛错不会炸掉 asyncio.gather 的其他分支，错误被封装成 ok=False 的信封向上传递。

async def _safe_run(self, inp, ctx):
    try:
        res = await self.run(inp, ctx)
        return res                        # ok=True
    except Exception as e:
        return ResultEnvelope.failure(e)   # 隔离，不向并发分支传播

7 · 三级降级策略

面对不稳定的 LLM 后端，系统设计了逐级升级的降级路径，保证任何情况下都不空跑。

L1 · 单步超时

某 SubTask 单步超过阈值（默认 90s）→ 用兜底 LLM 直答该 SubTask，其余正常。

L2 · 批量失败

同一 query 下 SubTask 失败比例 > 30% → 触发重规划，缩减 SubTask 数量后重试一轮。

L3 · 全局超时

整个 pipeline 超过上限（默认 10 分钟）→ 强制收敛，输出当前最优草稿并标 degraded=true。

8 · Agent 流水线

所有 Agent 继承 AbstractAgent[InT, OutT] 泛型基类，输入输出契约统一，trace_id 自动传播。

Agent输入 → 输出职责
Plannerquery → SubTask[]拆 3-8 个互补子问题，JSON Schema 校验，失败回退单任务
SearcherSubTask[] → Chunk[]每个 SubTask 调 Tavily，并发受控，正文不足时回退 Fetcher 抓取
ReaderChunk[] → KeyFacts三级压缩 + 写入共享记忆（含去重）
WriterKeyFacts → Report各章节并行写作，强制基于事实并标 [n] 引用
RedReport → Attack[]4 维结构化攻击 + 三层 JSON fallback
BlueAttack[] → Patch[]4 类 patch 动作，应用前过引用保留 invariant
真实运行

「RAG 检索增强生成」一题：Planner 拆 7 个 SubTask → Searcher 21 个 chunk → Reader 三级压缩（每段约 6000→3500 字符）→ Writer 生成带引用报告（含自动生成的 mermaid 流程图）→ Red 两轮共 14 个攻击 → Blue 共 12 个 patch 被采纳。

8b · 递归深挖（Recursive Deep-Dive）

对标 GPT-Researcher / dzhng deep-research 的 breadth/depth 调研后补的关键能力： 把单层 RAG 升级为多层 deep research。首轮检索完成后，对每个 SubTask 做"知识缺口分析"， 不足则生成更深入的追问子问题，递归检索-阅读-回填。

(per SubTask)"] --> GA["GapAnalyzer

组件

组件职责
agents/gap_analyzer.pyLLM 判断事实充分性，产出 ≤breadth 个更深入追问；严格 JSON，解析失败安全降级为"充分"（停止递归）
orchestrator/deepdive.py驱动递归、把新事实回填到父 SubTask、强制全局搜索预算上限

配额保护

关键设计

深挖会把 Tavily 用量放大至多 1 + depth×breadth 倍/SubTask，因此 默认关闭（depth=0），且必须经过 --deepdive-budget 全局硬上限。预算耗尽即优雅停止， 用已收集事实继续写作 —— 绝不会因失控扩展烧光月度配额。

CLI

# depth=2 两层追问，breadth=2 每层最多 2 个追问，预算 16 次搜索
dr-agent run "你的研究问题" --depth 2 --breadth 2 --deepdive-budget 16

真实验证

实测

「向量数据库 HNSW 索引」一题（depth=1, breadth=1, budget=4, max-results=3）： Planner 拆 6 个根 SubTask（6 次首轮搜索）→ deep-dive 发起 4 个追问（预算 4/4 用满后停止）→ 回填 293 个压缩事实。追问 query 往往命中比宽泛根 query 更丰富的来源。

诚实观察

预算 4 对 6 个根任务偏小，靠后的分支会被"饿死"（先到先用尽预算）。 合理预算应约 N × breadth。硬上限是配额安全网，不是质量调参旋钮。

9 · 三级语义压缩

Reader 阶段需要把搜回的 chunks（每篇约 1.5k 字符 × 多篇 × 多 SubTask）塞给 Writer，直接拼接会 token 爆炸。 三级压缩逐级精炼，同时保护关键信息不丢。

graph LR IN["原始 chunks"] --> L1["L1 Embedding 过滤
cosine < 0.45 丢弃"] L1 --> L2["L2 TextRank
句子图中心度 top-8"] L2 --> L3["L3 保护性保留
数字/年份/引号/缩写"] L3 --> OUT["压缩后 key facts"]
阶段粒度方法保留依据
L1chunk 级bge-small-zh 编码 → 与 SubTask query 的 cosine相似度 ≥ 0.45
L2句子级jieba 分词 + networkx PageRank 图中心度每 chunk top-8 关键句
L3句子级正则识别 \d+ / \d{4}年 / 引号 / [A-Z]{2,}含关键信息的句子强制保留

最终输出按 (is_protected DESC, score DESC) 排序，可选 token 预算截断。 实测压缩比约 40%（如某段 9739 → 6053 字符），L3 每段额外保护 37-150 个含关键信息的句子。

为什么要 L3

纯 TextRank 会丢掉"2017 年"「准确率 95%」这类低词频但高价值的事实句 （它们在图中心度上往往不高）。L3 用廉价正则把这类句子强制捞回来，是事实准确率能到 100% 的关键之一。

10 · 跨 Agent 共享记忆

用 SQLite + numpy 暴力 cosine 实现，而不是引入 Milvus/Qdrant —— 万级以下记录这是最优解。

存储结构

memory(
  id TEXT PRIMARY KEY, task_id, agent_id,
  text TEXT, embedding BLOB,   # float32 原始字节
  created_at REAL, is_duplicate INT, contradicts TEXT  # JSON id 列表
)

启动时全量 load 到内存矩阵，检索即 matrix @ query_vec 一次矩阵乘法。

向量检索

numpy 暴力 cosine，1.7k 条记录 < 50ms，万级 < 100ms。

预写去重

写入前查 top-1 相似度，cos > 0.92 直接跳过并标 duplicate。

矛盾检测

主语前缀相同但语义 cosine 反向超阈值 → 标记冲突进 review。

实测

单次研究任务后共享记忆累计约 2000+ 条事实；单 SubTask 去重命中 8-130 条不等， 证明跨 SubTask 信息确实存在大量重叠，去重有效降低了写作 prompt 的冗余。

11 · Red-Blue 对抗降噪

双 Agent 攻防：Red 找问题，Blue 修复，最多 K 轮（默认 K=2）。

Red：4 维结构化攻击

Red 输出严格 JSON（而非自然语言），每条攻击可执行：

{
  "attacks": [{
    "section_id": "sec-3",
    "type": "factual" | "logic" | "citation" | "completeness",
    "span": "<EXACT 子串，含 [n] 须原样保留>",
    "evidence": "<攻击依据>",
    "severity": 0.0..1.0,
    "suggested_action": "ADD | DELETE | MODIFY | VERIFY"
  }]
}

Blue：4 类 Patch 动作

动作语义必填
ADD在 span 之后追加 new_textnew_text
DELETE删掉 span—
MODIFY把 span 整体替换为 new_textnew_text
VERIFY保留原文，补充佐证（审计用）verification_evidence

引用保留 invariant（关键修复）

Prompt 约束 LLM 是概率事件，工程上必须有代码层的确定性兜底：

def _apply_patch(section, patch):
    if patch.action is PatchAction.MODIFY:
        span_cites = _citations_in(span)          # span 中的 [n]
        new_cites  = _citations_in(patch.new_text)
        if not span_cites.issubset(new_cites):
            return section, False   # 丢引用 → 拒绝该 patch
    if patch.action is PatchAction.DELETE:
        if 删除会使某 [n] 在本 section 孤立:
            return section, False   # 拒绝

11b · Multi-Critic 多角色评审（迭代）

对标 GPT-Researcher、Tongyi DeepResearch 等开源项目调研后，发现它们普遍没有对抗式自我修订。 本项目在 Red-Blue 基础上进一步迭代：把单个 Red（一个 prompt 兼顾 4 维度）拆成 3 个 persona 专精 critic 并行评审。

Critic专属职责攻击类型
FactChecker事实性错误 / 无依据断言factual
LogicReviewer内部矛盾 / 推理跳跃logic
CitationAuditor引用缺失 / 不对齐 / 来源质量低citation
graph TD R["报告草稿"] --> C1["FactChecker"] R --> C2["LogicReviewer"] R --> C3["CitationAuditor"] C1 --> M["双信号共识聚合"] C2 --> M C3 --> M M --> B["Blue 修复"] M -.span 去重.-> M M -.section 热点加权.-> M

双信号共识聚合

  • Span 级去重：近重复 span（cosine ≥ 0.82）合并为最高 severity 代表。
  • Section 级热点：persona 维度互斥，很少攻击同一 span；改为对「被 ≥2 个 critic 攻击的 section」加权（热点），孤立单 critic 攻击轻度降权。
设计主张

精度优先于召回：每个 persona 职责窄，攻击质量更高、更容易通过 Blue 的引用保留 invariant，胜过单个 Red 同时兼顾 4 维度的分心。

12 · 三层 JSON 解析 Fallback

LLM 输出 JSON 不是确定性的，即使设了 response_format={"type":"json_object"} 也偶尔被 markdown fence 或注释污染。三层兜底：

① direct

原始响应直接 json.loads，外加去 markdown fence + 正则提取 {...} 兜底。

② strict-retry

把错误响应注入 history，再发一条"只输出 JSON、无解释"的系统消息，温度降 0。

③ regex

从历次 raw 响应中正则抓 {...} 块，能解析就用。

实测

70+ 次真实 Red 调用中绝大多数走 direct， 偶尔触发 strict-retry 并成功修复，从未掉到第三层 —— 证明兜底设计在真实抖动下有效。

13 · 评测体系

三类指标 + 统计显著性，全部可重复。

ResearchBench

自建 11 领域 × 35 题，每题含 reference_facts（应命中的事实）、 forbidden_claims（已知错误，用于检测幻觉）、scoring_rubric（Judge 评分要点）。 覆盖：AI/ML 基础、RLHF、RL 算法、分布式系统、软件工程、数据库、网络安全、数据科学、生物医学、经济金融、历史地理。

① 规则指标（不依赖 LLM，最可靠）

指标计算方式方向
factual_accuracy报告句子与每条 reference_fact 的最大 cosine ≥ 0.55 → 命中越高越好
hallucination_rate报告句子与任一 forbidden_claim cosine ≥ 0.65 的比例越低越好
citation_coverage含 [n] 或 markdown 链接的句子占比越高越好

② LLM-as-Judge

独立 GPT-5.4，5 维度 × n_samples 自一致性；端点不可用时降级到 mimo 并标 self_bias_risk。

③ 统计显著性

  • Bootstrap (BCa) 95% CI：1000 次重采样，带偏差校正 z₀ + jackknife 加速因子 a。小样本/偏态分布比 percentile 更准。
  • Cohen's d 配对效应量：同一题在两配置下的得分配对计算，消除题目难度差异。|d|<0.2 可忽略，0.2-0.5 小，0.5-0.8 中，>0.8 巨大。
# BCa 偏差校正
z0 = norm.ppf((samples < point).mean())
# jackknife 加速因子
a = ((jack.mean()-jack)**3).sum() / (6 * ((jack.mean()-jack)**2).sum()**1.5)

15 · 翻车与修复（最有价值的工程故事）

第一次 35 题完整消融，Red-Blue 反而让指标退步了：

指标r0r2（初版）Δ
citation_coverage0.3930.320−0.073
factual_accuracy1.0000.996−0.004

根因定位

肉眼比对报告发现：Blue 在响应 completeness 攻击时，改写了含 [1][2] 的句子却忘了保留引用标记，引用被静默丢弃。

r0: 「GRPO 由 DeepSeek 提出 [1][2]，核心创新是组内相对优势 [3]」
r2(坏): 「GRPO 由 DeepSeek 提出，是一种针对长链推理优化的算法，核心创新是…」  ← [1][2][3] 没了

三层修复

  1. Red prompt：span 含 [n] 必须原样保留；completeness 攻击限 ≤ 2 条，优先 factual/citation。
  2. Blue prompt：MODIFY 的 new_text 必须保留 span 中所有 [n]。
  3. 代码 invariant（决定性兜底）：MODIFY 丢引用则拒绝该 patch；DELETE 会孤立引用则拒绝。

修复后（n=35 rerun，零额外 Tavily）

指标r0r2（修复后）ΔCohen's d
citation_coverage0.3930.472+0.079+1.31
hallucination_rate0.1640.158−0.005−0.06
factual_accuracy1.0001.000+0.000preserved
核心教训

代码 invariant 永远要兜住 LLM 的 prompt。 Prompt 遵循是概率事件， 引用保留这种硬约束必须用确定性代码守住最后一道关。这次翻车→定位→修复→数据回正的全过程，是整个项目最值得讲的工程故事。

其他真实教训

  • Tavily 缓存命中难：Planner 输出非确定，二跑生成的 SubTask query 与首跑不同，缓存几乎不命中 → 催生了"仅重跑 Red-Blue"的 rerun_red_blue.py 脚本（复用已存报告，零 Tavily）。
  • Judge 端点 502：外部依赖不可控 → fallback 到 mimo 并标注 self-bias，让开发不被 block 且数据可追溯。
  • meta tensor 报错：SentenceTransformer 在 worker 线程首次加载会报 meta tensor → 主线程 embedder.warmup() 预热解决。
  • Windows + Anaconda OpenMP 冲突：所有入口设 KMP_DUPLICATE_LIB_OK=TRUE。
